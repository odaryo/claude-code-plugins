  bp-audit の改善計画に基づいて Claude Code 設定を改善する。各変更をステップバイステップで
  ユーザー確認を取りながら適用し、検証結果を報告する。「改善を適用」「設定を最適化」
  「apply」「適用して」時に使用。
---

bp-audit で作成した改善計画に基づいて、設定を改善してください。

## Phase 1: 事前チェック

1. `git status` を実行し、作業ツリーの状態を確認
2. 未コミットの変更がある場合:
   - 「未コミットの変更があります。改善適用前にコミットすることを推奨します。続行しますか？」とユーザーに確認
   - ユーザーが中止を選択した場合は終了

## Phase 2: 改善計画の読み込み

1. `$ARGUMENTS` でパスが指定されている場合はそのファイルを読み込み
2. 指定がない場合、`.ai/setting_reports/*/*/bp-plan*.md` を Glob で検索
3. 各ファイルの frontmatter から `created_at` を読み取り、最新のファイルを選択
4. frontmatter がないファイルはディレクトリ名から日付を復元（フォールバック）
5. 計画が見つからない場合は「先に `/bp-audit` を実行してください」と案内

計画を読み込んだら、アクション一覧をユーザーに表示:
```
改善計画: {計画ファイルパス}
アクション数: N件

[P1] {タイトル} — {対象ファイル}
[P2] {タイトル} — {対象ファイル}
...

優先度順に1件ずつ適用します。各変更は個別に確認を取ります。
```

## Phase 3: アクションの適用

優先度順（P1 → P2 → ...）に各アクションを処理:

### 3a. 変更内容の表示

対象ファイルが存在する場合:
- ファイルの該当部分を読み込み
- before/after の diff 形式で変更内容を表示

対象ファイルが存在しない場合（新規作成）:
- 作成するファイルの全内容を表示

### 3b. ユーザー確認

「この変更を適用しますか？」とユーザーに確認。選択肢:
- **適用する** → 3c へ
- **スキップ** → 次のアクションへ
- **変更内容を修正して適用** → ユーザーの修正指示を受けて調整後に適用
- **残り全てスキ���プ** → Phase 4 へ

### 3c. 変更の適用

- 既存ファイルの変更: Edit ツールを使用
- 新規ファイルの作成: Write ツールを使用
- `.gitignore` へのエントリ追加: 既存内容に追記

### 3d. 検証

適用後:
1. ファイルを Read で読み戻し、変更が正しく反映されているか確認
2. JSON ファイル（settings.json 等）の場合は `jq .` で構文検証
3. ��題があれば即座にユーザーに報告

### 安全ルール（厳守）

- **APIキーやトークンの値そのものを設定ファイルに書き込まない**
- **破壊的コマンド（`rm -rf`, `git push --force` 等）を実行しない**
- **グローバル設定（`~/.claude/settings.json`）への変更は「参考提案」セクションのものに限り、追加の確認を取る**
- **各変更は必ずユーザー確認を経てから適用する**
